# Timetriq UI Guidelines

Version: 1.0

---

# Purpose

This document defines the UI implementation standards for Timetriq.

Its purpose is to ensure every interface generated by AI follows a consistent design language, component structure, accessibility standard, and user experience philosophy.

When generating UI, always follow these guidelines before introducing new layouts or components.

---

# Design Philosophy

Timetriq is a productivity application.

The interface should help users complete work efficiently, not impress them with unnecessary visual effects.

Every screen should prioritize:

- Clarity
- Consistency
- Simplicity
- Productivity
- Accessibility

The best UI is one that requires little explanation.

---

# Design Principles

Every screen should be:

- Clean
- Minimal
- Predictable
- Responsive
- Accessible

Avoid unnecessary decoration.

Avoid excessive animations.

Avoid visual clutter.

---

# Technology Standards

Frontend Framework

- React

UI Library

- shadcn/ui

Styling

- Tailwind CSS

Icons

- Lucide React

Charts

- Recharts

Calendar

- FullCalendar

Do not introduce additional UI frameworks unless explicitly approved.

---

# Layout Rules

All authenticated pages should follow the same structure.

```
------------------------------------------------
 Header
------------------------------------------------
 Sidebar | Main Content
         |
         |
         |
------------------------------------------------
```

The layout should remain consistent across the application.

---

# Navigation

Primary navigation includes:

- Dashboard
- Tasks
- Calendar
- Reports
- Settings

Navigation should always indicate the active page.

On smaller screens, the sidebar should collapse into a mobile-friendly navigation.

---

# Page Structure

Every page should contain:

1. Page Title
2. Short Description
3. Primary Action
4. Filters/Search (if applicable)
5. Main Content
6. Supporting Information

Do not rearrange this structure without a strong usability reason.

---

# Component Hierarchy

Prefer composition over large components.

Example

```
TasksPage

├── PageHeader

├── TaskFilters

├── TaskTable

│   ├── TaskRow

│   └── TaskActions

└── TaskDialog
```

Avoid monolithic page components.

---

# Component Rules

Components should:

- Have one responsibility
- Be reusable
- Accept typed props
- Avoid business logic
- Avoid direct API calls

Business logic belongs outside components.

---

# Forms

Use:

- React Hook Form
- Zod validation

Forms should provide:

- Immediate validation feedback
- Clear labels
- Helpful error messages
- Loading indicators during submission

Avoid placeholder text as the only label.

---

# Tables

Tables should support:

- Sorting
- Filtering
- Searching
- Pagination

Column headers should remain clear and concise.

Avoid overcrowding tables.

---

# Dashboard

Dashboard priority:

1. Summary Metrics
2. Today's Work
3. Capacity Overview
4. Upcoming Tasks
5. Recent Activity

Users should understand their workload within a few seconds.

---

# Calendar

Calendar should display:

- Planned workload
- Capacity
- Tasks

Overloaded days should be visually distinguishable.

Do not overload the calendar with unnecessary information.

---

# Cards

Cards should:

- Represent one concept
- Use consistent spacing
- Include concise information
- Avoid excessive actions

---

# Dialogs

Use dialogs for:

- Creating tasks
- Editing tasks
- Confirming destructive actions

Dialogs should remain focused on a single workflow.

---

# Buttons

Button hierarchy:

Primary

Main action.

Secondary

Supporting action.

Destructive

Delete or archive actions.

Do not use destructive styles for non-destructive actions.

---

# Search

Search should:

- Respond quickly
- Support partial matches
- Ignore case
- Clearly indicate when no results are found

---

# Empty States

Every empty state should include:

- Clear explanation
- Helpful illustration (optional)
- Primary action

Example:

"No tasks found."

Button:

"Create Task"

---

# Loading States

Prefer skeleton loaders for page content.

Use button loading indicators during actions.

Avoid blocking the entire page unless required.

---

# Error States

Errors should:

- Explain what happened
- Suggest the next step
- Avoid technical terminology

Example:

"Unable to load tasks. Please try again."

---

# Notifications

Use toast notifications for:

- Successful saves
- Updates
- Deletions
- Minor errors

Do not interrupt users with unnecessary dialogs.

---

# Icons

Use Lucide React icons consistently.

Icons should support text rather than replace it.

Avoid mixing icon libraries.

---

# Typography

Maintain a clear hierarchy.

Levels:

- Page Title
- Section Heading
- Card Title
- Body Text
- Caption

Avoid inconsistent font sizing.

---

# Spacing

Use consistent spacing throughout the application.

Prefer whitespace over crowded layouts.

Maintain consistent padding and margins.

---

# Colors

Use color intentionally.

Primary

Brand identity and primary actions.

Success

Completed or successful actions.

Warning

Potential issues.

Error

Failures and destructive actions.

Information

General feedback.

Do not rely on color alone to communicate meaning.

---

# Responsive Design

Support:

- Desktop
- Tablet
- Mobile

Layouts may adapt.

Functionality must remain consistent.

---

# Accessibility

Every interface must support:

- Keyboard navigation
- Focus indicators
- Screen readers
- Semantic HTML
- Accessible color contrast
- Proper form labels

Accessibility is mandatory.

---

# Performance

Optimize UI performance by:

- Lazy loading pages
- Memoizing expensive components
- Virtualizing long lists
- Avoiding unnecessary re-renders

Measure performance before optimizing.

---

# Animation

Animations should:

- Be subtle
- Improve usability
- Never delay user actions

Avoid decorative animations.

---

# AI UI Generation Rules

Before creating a new component:

Check whether a similar component already exists.

Prefer reusing existing components.

Maintain consistent styling.

Follow the existing design system.

Do not introduce inconsistent patterns.

---

# Screen Review Checklist

Before considering a screen complete verify:

- Consistent layout
- Responsive behavior
- Accessible controls
- Proper spacing
- Correct typography
- Clear hierarchy
- Helpful feedback
- No visual clutter

---

# Future Enhancements

The UI should accommodate future features without major redesign.

Examples:

- Dark Mode
- Team Workspaces
- AI Assistant Panel
- Drag-and-Drop Planning
- Real-Time Collaboration
- Mobile Support

Design for extensibility from the beginning.

---

# Final Principle

Every screen should feel like part of the same product.

Users should be able to move between pages without relearning interactions, layouts, or visual patterns.

A consistent interface builds trust, reduces cognitive load, and allows users to focus on planning and completing work rather than understanding the software.
